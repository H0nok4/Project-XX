# Project-XX 重构路线图

## 1. 当前阶段判断

截至 2026-03-09，项目已经完成了“局外配置 -> 局内战斗 -> 撤离 / 死亡 -> 回写局外”的主闭环。

已落地的关键能力：

- 局外仓库、武器柜、战斗背包、安全箱、特殊装备
- 商店购买与出售
- 局内 FPS 战斗、体力、姿态、速度比例
- 部位伤害、护甲、状态效果
- AI archetype、噪声感知、近战 / 远程攻击
- 随机 Loot、尸体搜刮、门、破坏物
- 原型关卡搭建工具

因此接下来的问题已经不是“先做功能还是先搭框架”，而是：

- 哪些地方继续堆功能会迅速失控
- 哪些地方值得在保持原型节奏的前提下开始拆分

---

## 2. 当前主要风险

### 2.1 巨型控制器

风险最大的三个类：

- `PrototypeFpsController`
- `PrototypeBotController`
- `PrototypeMainMenuController`

这三者都已经跨越了多个职责边界。

### 2.2 存档仍是定义级

当前 Profile 仍以：

- `itemId`
- `quantity`
- `weaponId`

为主。

这会限制后续：

- 武器实例状态
- 护甲耐久跨局保留
- 改枪和个体装备差异

### 2.3 UI 仍偏原型

当前主菜单、仓库、商店、背包、搜刮、结算大量依赖 IMGUI。

这对快速验证很好，但对下列内容不友好：

- 多容器拖拽
- 更复杂布局
- UI 动画与视觉风格
- 后续制作质量

### 2.4 原型场景与正式内容还没分家

当前 Builder 仍可以重建核心场景和默认资产。

优点：

- 快速重置原型

风险：

- 正式内容容易被覆盖
- 内容制作与工具脚本耦合

---

## 3. 建议的下一阶段顺序

## 阶段 A：清理局外 UI 与资料流

### 目标

- 稳住仓库 / 商店 / 战斗配置逻辑
- 降低 `PrototypeMainMenuController` 继续膨胀的风险

### 建议动作

- 拆出：
  - `MetaInventoryPresenter`
  - `MetaMerchantPresenter`
  - `MetaLoadoutPresenter`
- 保留 `PrototypeMainMenuController` 只做页面路由和顶层协调
- 把当前乱码 / 中英混合文案顺带彻底清理

### 进入条件

- 继续增加更多商店、任务、钥匙、交易规则之前

### 完成标准

- 主菜单控制器不再直接承载全部局外 UI 细节
- 页面之间的状态共享不再依赖单个大类字段

---

## 阶段 B：做实例级物品 / 武器 / 护甲

### 目标

- 从“定义级存档”升级到“实例级存档”

### 需要解决

- 武器当前弹匣余弹
- 武器耐久
- 护甲剩余耐久
- 后续改枪 / 特殊词条 / 个体差异

### 建议动作

- 引入：
  - `SavedItemInstanceDto`
  - `SavedWeaponInstanceDto`
  - `SavedArmorInstanceDto`
- 扩展 `ItemInstance`
- 区分：
  - 定义资产
  - 运行时实例
  - 存档 DTO

### 进入条件

- 计划开始做武器实例状态带出
- 计划开始做护甲耐久跨局保存

### 完成标准

- 撤离后可以保留“同一种枪，但状态不同”的实例

---

## 阶段 C：重做局内多容器 UI

### 目标

- 把局内背包、尸体、箱子、安全箱之间的转移做成真正可扩展的多容器交互

### 当前缺口

- 还没有完整的拖拽式多容器管理
- 安全箱在局内虽然存在，但交互体验还不完整

### 建议动作

- 先定义统一容器视图模型
- 再把：
  - 主背包
  - 安全箱
  - 特殊装备
  - 箱子
  - 尸体 Loot
  接到同一套 UI

### 进入条件

- 要做 Tarkov-like 的局内整理体验

### 完成标准

- 玩家可在局内把物品从尸体 / 箱子拖到主背包或安全箱

---

## 阶段 D：拆分玩家控制器

### 目标

- 控制 `PrototypeFpsController` 的复杂度

### 建议拆分

- `PrototypePlayerMovementController`
- `PrototypePlayerWeaponController`
- `PrototypePlayerMedicalController`
- `PrototypePlayerHudPresenter`

### 原则

- 先拆运行时职责
- 不先推翻输入层
- `PrototypeFpsInput` 继续保留为统一输入入口

### 进入条件

- 继续增加体位、瞄准、换弹细节、后坐力、武器晃动等玩法前

### 完成标准

- 任一玩家功能迭代不再需要理解整个大控制器

---

## 阶段 E：拆分 AI 控制器

### 目标

- 控制 `PrototypeBotController` 的继续膨胀

### 建议拆分

- `BotPerception`
- `BotLocomotion`
- `BotCombat`
- `BotLootOnDeath`
- `BotArchetypeConfig`

### 进入条件

- 计划增加更多敌人 archetype
- 计划增加特殊行为，如扑击、投掷、召唤、团队协同

### 完成标准

- 改攻击节奏时不需要同时碰感知与移动实现

---

## 阶段 F：从 IMGUI 迁移

### 目标

- 把原型 UI 迁移到更适合长期维护的实现

### 建议顺序

1. 主菜单首页与仓库页
2. 商店页
3. 局内背包 / 搜刮页
4. 战局结算页
5. HUD

### 进入条件

- 要做更像正式产品的 UI
- 要做拖拽、动画、复杂信息层级

### 完成标准

- UI 表现和逻辑状态完全分离

---

## 阶段 G：原型场景与正式内容拆分

### 目标

- 避免 Builder 继续覆盖未来正式内容

### 建议做法

- 保留：
  - `MainMenu_Prototype`
  - `SampleScene_Prototype`
- 新建：
  - `MainMenu_Production`
  - `RaidBlockout_Production`
  - 后续正式关卡场景

### 进入条件

- 美术、关卡、策划开始长期手工摆场景时

### 完成标准

- Builder 只负责原型
- 正式内容不再依赖重建脚本

---

## 4. 当前不建议优先做的重构

以下内容现在不建议抢在前面：

- 先拆 asmdef 再继续做功能
- 先完全重写 AI 再扩敌人
- 先重写全部 UI 再修数据结构
- 先推翻现有 Builder 再做正式关卡原型

原因：

- 这些动作会显著打断当前可玩的原型节奏
- 但对近期“继续做内容验证”的直接收益不高

---

## 5. 现阶段最合理的优先级

如果只选一个近期主方向，建议是：

1. 实例级物品 / 武器 / 护甲存档
2. 局内多容器交互 UI
3. 主菜单 / 仓库 / 商店 UI 拆分

原因：

- 这三项会直接决定 Tarkov-like 核心循环能否继续深入
- 比继续堆更多敌人或更多枪，更能决定系统后续是否可持续扩展

---

## 6. 完成判断

当以下条件成立时，可以认为当前原型已进入“第二阶段”：

- 局外系统不再只有单个大 UI 控制器
- 局内物品和装备已能实例化保存
- 安全箱 / 背包 / 尸体 / 容器之间能完整转移
- 玩家与 AI 控制器职责开始拆分

到那时，项目就可以从“功能验证原型”进入“可持续扩展原型”。
