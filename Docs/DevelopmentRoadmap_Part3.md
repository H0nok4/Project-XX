# Project-XX AI开发路线图（执行拆分）

## 1. 文档目的

本文件不是替代 [DevelopmentRoadmap_Part2.md](./DevelopmentRoadmap_Part2.md)，而是把其中的中后期阶段继续拆成更适合排期和 AI 落地的执行批次。

重点解决三个问题：

- 大阶段先做什么、后做什么
- 哪些内容适合拆成一批一批交给 AI 落地
- 哪些内容现在不该提前做，避免范围失控

---

## 2. 拆分原则

### 2.1 先做底座，再做内容量

- 先完成数据结构、运行时服务、状态回写
- 再做交互UI、场景点位、演出表现
- 最后再扩内容量和美术表现

### 2.2 每个批次只解决一个核心问题

每个 AI 开发批次应尽量满足：

- 只围绕一个运行时真相来源展开
- 涉及的核心脚本不超过 5-8 个
- 有明确的手动验证路径
- 必须附带文档更新

### 2.3 不允许跳过状态真相

下列系统必须始终保持单一真相来源：

- Profile / 世界状态：`PrototypeProfileService` / `WorldStateService`
- 任务状态：`QuestManager`
- 战斗属性汇总：`CharacterStatAggregator`
- 剧情触发：`NarrativeDirector`
- 字段归属、回写时机、存档迁移规则以 [StateOwnershipAndPersistenceRules.md](./StateOwnershipAndPersistenceRules.md) 为准

### 2.4 Builder 和场景工具不是业务真相

- 地图摆放工具只负责配置和批量放置
- 任务逻辑、商人解锁、剧情状态不能只存在于编辑器工具代码里
- 商人等级 / 信誉 / 解锁状态 / 库存刷新职责边界以 [MerchantProgressionMatrix.md](./MerchantProgressionMatrix.md) 为准

### 2.5 前置治理必须先落地

- 在进入成长、世界状态、任务链扩展前，先完成存档 schema 版本化与迁移基线
- 在基地正式化前，先定义 `MainMenu` 到 `BaseScene` 的入口迁移与兼容层
- 这两项虽然不是“内容功能”，但属于后续所有 AI 批次的约束前提

## 3. 里程碑总览

### M0：前置治理与入口迁移

对应阶段：

- `DevelopmentRoadmap.md` 阶段0.4 - 0.5
- `DevelopmentRoadmap_Part2.md` 阶段2.6

目标：

- 在继续扩成长、世界状态和基地功能前，先稳定存档治理与局外入口职责

完成标志：

- 新旧档迁移路径、备份与诊断策略明确
- `BaseScene` 可作为正式局外入口，`MainMenu` 退化为启动壳 / 调试入口

### M1：角色成长底座

对应阶段：

- 阶段4.1 - 4.5

目标：

- 让等级、属性、技能树、装备词条、派生属性形成统一结算链

完成标志：

- 玩家成长数据可以存档
- 装备与技能可以共同影响战斗数值
- 基地内可以查看成长与构筑结果

### M2：世界状态与局内中立交互底座

对应阶段：

- 阶段5.1 - 5.2

目标：

- 为地图商人、地图任务NPC、剧情节点建立稳定的状态与交互基础

完成标志：

- 世界状态可以跨局保存
- 战斗地图中可稳定放置并交互中立NPC

### M3：地图商人、任务NPC与任务链

对应阶段：

- 阶段5.3 - 5.6

目标：

- 让“地图中解锁商人”和“地图内任务链”真正成为玩法的一部分

完成标志：

- 至少一名地图商人可通过任务解锁
- 至少一条战斗地图任务链可完整跑通

### M4：剧情表现底座

对应阶段：

- 阶段6.1 - 6.6

目标：

- 让剧情系统具备对话、CG、视频、过场、触发和回放能力

完成标志：

- 至少一段基地剧情和一段局内剧情可稳定播放

### M5：基地正式化与制作经济

对应阶段：

- 阶段7.1 - 7.5

目标：

- 把基地从功能菜单升级为长期成长中枢

完成标志：

- 玩家可在基地完成制作、强化、交易、成长、接任务和出击准备

### M6：内容扩展样板

对应阶段：

- 阶段8.1 - 8.5

目标：

- 先做出“完整内容样板”，而不是立刻无限铺量

完成标志：

- 2-3 张正式地图
- 至少 1 章主线 + 若干支线
- 至少 1 套地图商人 / 任务NPC / Boss 联动内容

### M7：优化与发布准备

对应阶段：

- 阶段9.1 - 9.5

目标：

- 从“可玩”推进到“稳定、可持续调试、可准备正式版”

完成标志：

- 关键性能、平衡、存档、QA工具都达到稳定状态

## 4. 里程碑详细拆分

### 4.0 M0：前置治理与入口迁移

详细设计与任务拆分：

- 技术设计见 [M0FoundationTechnicalDesign.md](./M0FoundationTechnicalDesign.md)
- 第一批任务见 [M0FirstBatchTasks.md](./M0FirstBatchTasks.md)

#### 建议拆分顺序

1. 存档 schema 与迁移基线
2. 局外入口迁移与兼容层

#### AI任务包

#### M0-A：存档Schema版本化与迁移基线

**目标**

- 给 `ProfileData`、`WorldStateData` 和后续可扩展子结构建立统一版本化、迁移、备份与校验入口

**交付物**

- `saveVersion` / `schemaVersion` 字段约定
- 存档迁移接口与执行入口
- 备份与诊断输出规则

**完成标准**

- [ ] 旧档升级到新结构的路径清晰
- [ ] 迁移失败时可保留备份并输出诊断
- [ ] 后续 M1 / M2 新增字段不需要各自发明一套迁移逻辑

#### M0-B：局外入口迁移与兼容层

**目标**

- 明确 `MainMenu` 与 `BaseScene` 的职责边界，避免正式局外系统双维护

**交付物**

- `MetaEntryRouter`
- `MainMenu` -> `BaseScene` 迁移策略
- 调试入口兼容方案

**完成标准**

- [ ] 正式局外入口以 `BaseScene` 为准
- [ ] `MainMenu` 仅保留启动壳 / 调试用途
- [ ] 不再为同一套局外功能维护两份入口逻辑

#### 本里程碑不要提前做

- 不要先把更多业务面板继续堆进 `MainMenu`
- 不要等多个系统都接入后再补存档迁移基线

### 4.1 M1：角色成长底座

#### 建议拆分顺序

1. 先做成长存档结构
2. 再做属性系统
3. 再做技能树
4. 然后做属性汇总器
5. 最后补成长 UI

#### AI任务包

#### M1-A：成长存档结构与资料入口

**目标**

- 在现有 Profile 基础上增加等级、经验、属性点、技能点、已解锁节点等数据

**交付物**

- `PlayerProgressionData`
- `PrototypeProfileService` 扩展字段
- 存档迁移逻辑

约束：

- 字段归属、迁移入口、失败回滚遵循 [StateOwnershipAndPersistenceRules.md](./StateOwnershipAndPersistenceRules.md)

**完成标准**

- [ ] 新档可正常初始化成长数据
- [ ] 旧档可迁移到新结构
- [ ] 不影响现有局外/局内闭环

#### M1-B：基础属性系统

**目标**

- 建立基础属性与派生属性的对应关系

**交付物**

- `PlayerAttributeType`
- `PlayerAttributeSet`
- `PlayerDerivedStats`

**完成标准**

- [ ] 属性分配可生效
- [ ] 派生属性可被读取
- [ ] 属性变化后可即时刷新

#### M1-C：技能树定义与解锁

**目标**

- 建立技能树数据和解锁规则，不先追求复杂表现

**交付物**

- `SkillNodeDefinition`
- `PlayerSkillTree`
- 技能树解锁服务

**完成标准**

- [ ] 可以解锁节点
- [ ] 前置条件生效
- [ ] 解锁状态可保存

#### M1-D：属性汇总器接入战斗系统

**目标**

- 把成长、装备、词条、技能统一汇总到一个运行时入口

**交付物**

- `CharacterStatAggregator`
- `PrototypeFpsController` / `PrototypeUnitVitals` 接入

**完成标准**

- [ ] 面板数值与实际战斗数值一致
- [ ] 装备变化与成长变化都能刷新
- [ ] 不再到处散落派生属性计算

#### M1-E：成长面板

**目标**

- 在基地中提供可视化成长入口

**交付物**

- 成长页 / 角色页
- 构筑摘要面板

**完成标准**

- [ ] 可查看等级、属性、技能树
- [ ] 可理解当前Build主要方向

#### 本里程碑不要提前做

- 不要先做复杂的技能树动画和视觉包装
- 不要先做大量技能内容池
- 不要让 UI 直接持有成长真相数据

### 4.2 M2：世界状态与局内中立交互底座

#### 建议拆分顺序

1. 世界状态结构
2. 世界状态存档
3. 战斗地图中立交互区
4. 中立NPC基础交互

#### AI任务包

#### M2-A：WorldStateData 与存档接入

**目标**

- 让商人解锁、任务链阶段、剧情标记具备统一保存结构

**交付物**

- `WorldStateData`
- `WorldStateService`
- `PrototypeProfileService` 世界状态读写

约束：

- `WorldStateData` 只记录解锁、显隐、剧情标记、章节结果，不记录原子任务目标计数
- 详细边界遵循 [StateOwnershipAndPersistenceRules.md](./StateOwnershipAndPersistenceRules.md)

**完成标准**

- [ ] 世界状态可跨局保存
- [ ] 任务链与剧情标记不再散落在多个系统里

#### M2-B：中立交互区规则

**目标**

- 建立地图商人和任务NPC可共用的交互区规则

**交付物**

- `RaidNpcZone`
- `RaidNpcInteractor`
- 输入冻结 / 恢复逻辑

**完成标准**

- [ ] 战斗地图中可与中立NPC稳定交互
- [ ] 交互期间玩家控制切换正确

#### M2-C：世界状态驱动显隐

**目标**

- 让同一张地图内的商人/NPC能根据解锁状态显隐或切换行为

**交付物**

- `WorldStateCondition`
- NPC 显隐与可交互判定层

**完成标准**

- [ ] 未解锁时不可交互
- [ ] 解锁后进入地图可见并可交互

#### 本里程碑不要提前做

- 不要先做复杂地图商店UI
- 不要先做剧情演出
- 不要先做大量地图NPC内容

## 5. 当前最推荐的实际开工顺序

如果你准备马上进入下一轮开发，建议按下面顺序排：

1. `M0-A` 存档Schema版本化与迁移基线
2. `M0-B` 局外入口迁移与兼容层
3. `M1-A` 成长存档结构与资料入口
4. `M1-B` 基础属性系统
5. `M1-C` 技能树定义与解锁
6. `M1-D` 属性汇总器接入战斗系统
7. `M1-E` 成长面板
8. `M2-A` 世界状态与存档接入
9. `M2-B` 中立交互区规则
10. `M3-A` 地图商人框架
11. `M3-B` 地图任务NPC框架
12. `M3-C` 局内目标与触发器

这个顺序的原因是：

- 先把存档治理和入口迁移定住，后面每个系统接入都不需要重复返工
- 再把 RPG 成长底座立住，装备等级、品质、词条、技能才有稳定接入点
- 技能树应先于属性汇总器完成最小可用接入，否则汇总器会先做半套再返工
- 再做世界状态，不然后面的地图商人和任务链状态无法落盘
- 最后再进入地图商人与任务NPC，不会边做边返工底层

## 6. 适合交给 AI 的单次任务粒度

建议你后面每次给 AI 的任务，尽量控制在以下粒度：

- 一个核心服务或一个核心运行时对象
- 一次只跨一个主要领域，例如“成长”或“任务”，不要同时改“成长 + 剧情 + 商店”
- 如果涉及 UI，先把运行时和数据层做完，再补 UI
- 每次都要求 AI 同步更新文档和手动验证步骤

推荐的任务描述模板：

```text
目标：
边界：
涉及文件：
不要修改：
验收标准：
手动验证步骤：
```

## 7. 当前不建议提前做的内容

- 不建议现在就做完整大规模剧情资源生产
- 不建议现在就做大量正式地图铺量
- 不建议现在就把所有原型 UI 一次性重写成正式UI
- 不建议在世界状态和属性汇总器落地前做复杂地图商人和复杂任务链
- 不建议在经济未稳定前开放过强的词条重铸和装备强化
